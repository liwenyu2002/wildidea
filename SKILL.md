# 🌐 WildIdea — 狂野想法

## 核心哲学

> 旧路径（禁止）：用户问题 → 常规答案 → 找跨域类比包装
> 新路径（强制）：用户问题背景压缩 → 三线注入远域 → 远域先自己说话 → 外来原则 → 反向重写用户问题 → 翻译回来

**防圆回纪律：**
1. ❌ 禁止先给常规答案再找跨域包装
2. ❌ 禁止把远域降级为比喻（"XX像YY，所以"）
3. ❌ 禁止因"看起来不相关"早期淘汰随机领域
4. ❌ 禁止搜不到时编造搜索结果（方法B搜不到就跳过）
5. ✅ 每条建议通过"删除远域测试"——去掉远域还能自然想到的，删除

## 领域来源架构

| 方式 | 数量 | 参考文件 |
|------|------|---------|
| 🔵 预置远域范式 | 3 | `references/paradigms.md`（17类，偏向后9类更野的域） |
| 🟢 种子库随机 | 2 | `references/mao-seeds.md`（50条毛选） |
| 🟠 随机组字+联网撞 | 1 | `scripts/search_char.py`（字库+搜索） |

## 核心流程

### Step 0 — 常规答案隔离
列出最易滑入的3-5条常规路径（如"加一个VAE+ODE"、"堆Transformer"），暂时隔离。

### Step 1 — 背景压缩（不求解）
只记录用户领域、问题、想突破的惯性。禁止提炼"核心张力"或倾向答案。

### Step 2 — 三线并行注入异质领域
三条线同时跑：
- 🔵 **预置远域**：加载 `skill_view('wildidea', 'references/paradigms.md')`，扫描全部17类，偏向后9类（7.9-7.17，它们离产品/商业更远），选3个不同大类
- 🟢 **种子库**：运行 `python3 {skill_dir}/scripts/pick_seed.py` 随机抽2个种子 → 解构抽象结构 → 联想领域。**禁止**根据用户问题挑选种子。
- 🟠 **随机组字**：运行 `python3 {skill_dir}/scripts/search_char.py` 取字→Tavily搜→提取真实领域。搜不到标注🚫并跳过。

### Step 3 — 远域自洽建模（先让它自己说话，禁止翻译）
对每个选中的远域（共6个），**先用远域自己的核心术语**描述这个域的核心机制（2-3句）。绝对禁止使用用户领域的任何词语。

> ✅ 示例（远域=蜜蜂舞蹈）：
> "蜜蜂通过摇摆舞传递食物源方向和距离——舞动角度=太阳偏角，振翅时长=距离。其他工蜂通过触觉和嗅觉确认信息，不需要全局地图。"
>
> ❌ 错误（提前翻译）：
> "蜜蜂用舞蹈传递信息，就像用户用口碑推荐产品……"

### Step 4 — 外来原则提取
从Step 3的描述中提取2-3条该远域的核心原则/机制/约束。仍然用远域术语。禁止混合用户术语。

### Step 5 — 反向重写
把用户的问题用远域的术语重新描述一次（"如果你是一个[远域系统]，你怎么看这个问题？"）。**仍然用远域术语。**

### Step 6 — 翻译回用户领域
把Step 5的结果翻译成用户能听懂的语言。**此时才第一次允许使用用户领域术语。**

### Step 7 — 🌪️ 舒适度检查
问自己："如果这条建议发到产品评审群里，会有人皱眉反对吗？"

- 答案是"不会，挺合理的" → ❌ 不够wild，回到Step 3重新选范式映射
- 答案是"这他妈谁会同意……但好像有点道理" → ✅ 通过

### Step 7.5 — 🔍 联网验证 + 循环抽卡
对每条通过舒适度检查的映射，执行联网搜索验证：

1. **构造搜索词**：[核心动作/建议] + [用户领域]
2. 用 `web_search` 搜索（Tavily或Brave）
3. 如果结果显**该建议已被同行做过、已发布、是已知方案** → 🚫 ban掉
4. 回到该远域**剩余的范式池**，重新选一个未用过的范式映射
5. 重新跑Step 3→7.5
6. 每个远域最多重试3次。所有范式用光都被ban → 标注🚫并跳过该远域
7. 每个远域最终通过验证的映射标注验证结果（如 `✅ 未找到已实现案例` 或 `🚫 全部范式用尽，跳过`）

### Step 8 — 综合输出

输出格式：

| # | 来源 | 这个远域自己怎么运行的 | 翻译到你的领域 | 验证 |
|---|------|----------------------|--------------|------|
| 1 | 🔵 7.N [域]·[范式] | 用远域自己的术语描述它怎么运行（2-3句，禁止你的领域术语） | 映射到你的领域的一个具体动作/产品策略 | ✅ 未找到已实现 |
| 2 | 🟢 种子#[#]「种子原文」 | 同上 | 同上 | ✅ 未找到已实现 |
| 3 | 🟠 随机组字「词」→[域] | 同上 | 同上 | 🚫 跳过 |

**输出纪律（违反即不合格）：**
- ❌ 禁止在最终输出中出现Step 3/4/5的推导过程——只展示Step 3的远域描述 + Step 6的翻译结果 + 验证结果
- ❌ 如果某条映射需要3段话才能说清 → **压缩成1句"它怎么运行的"+"1句翻译"**
- ✅ 隔离区标注被ban的常规答案 和 前轮产出（二次迭代时）
- ✅ 通过在底部追加 `🌪️` 通过条数 / `🔍` 验证结果 / `🚫` ban掉条数

## 质量红线

1. 至少3个不同来源领域
2. 至少1个来自随机种子或随机组字
3. 每条主建议通过删除远域测试
4. 不允许只说"这像XX"
5. 不允许搜索失败时编造结果
6. 最终建议至少有一条改变了用户问题定义
7. **🌪️ 舒适度检查不通过（太合理）→ 不放行**
8. **🔍 联网验证被ban（已有同行做过）→ 重抽重做，不放行**

## 二次迭代（迭代深化模式）

当一轮 WildIdea 输出后，用户可能要求「以相同问题再挖一层」：

### 触发条件
用户说「把产出一起ban掉重跑」「迭代」「再挖一层」「二次迭代」「重跑」

### 流程

1. **构建双层隔离区**：
   - 第一层（round-0 ban）：该问题的常规答案隔离区（Step 0 标准输出）
   - 第二层（round-N ban）：上一轮 WildIdea 输出的所有6条结果（每条的「所以你试试」核心动作）

2. **重跑全部三线**（禁止复用上轮的种子/领域）：
   - 🔵 预置远域：重新选3个不同大类（与上轮不重复）
   - 🟢 种子库：重新 `pick_seed.py`（结果必然不同）
   - 🟠 随机组字：`search_char.py` 再跑一次（随机组合+Tavily）

3. **内部映射**（Step 3→7）：在全新领域上做映射，强制避开所有被ban的方向

4. **输出**：海报包含双层隔离区（两个 `.row`），第一行「首轮隔离」+ 第二行「前轮产出已ban」
   - 第二层标签使用 `.tag-ban2` CSS 类（暗红色边框划掉，视觉上区别于第一层 `.tag`）

### 迭代纪律
- ❌ 禁止回到上轮已经ban掉的方向（「xx换个说法」）
- ❌ 禁止在用户没说「够深了」前停止迭代
- ✅ 用户说「这个方向可以展开」→ 保持当前迭代但深挖某条
- ✅ 用户说「这个太圆了」→ 删掉那条，重抽种子重做

## 视觉输出流程（当用户要求海报/图片时）

1. **加载海报模板**：`skill_view('wildidea', 'templates/poster.html')` — 获得带占位符的完整HTML骨架
2. **参考配色方案**：`skill_view('wildidea', 'references/poster-palettes.md')` — 根据轮次/主题选色
3. **替换占位符**（以下所有占位符必须在生成前替换）：

   | 占位符 | 说明 | 示例值 |
   |--------|------|--------|
   | `{TITLE}` | 标题文字 | 新能源汽车 · 如何建立竞争优势 |
   | `{BADGE_TEXT}` | 徽章标签（轮次/版本） | WILDIDEA · ROUND 2 |
   | `{BADGE_COLOR}` | 徽章渐变色Hex | #06b6d4 或 #f59e0b |
   | `{BADGE_ACCENT}` | 高亮色Hex | #22d3ee 或 #fbbf24 |
   | `{GRADIENT}` | 标题文字渐变色CSS | `linear-gradient(135deg, #38bdf8, #a78bfa)` |
   | `{FOCUS}` | 焦点说明文字 | 新品牌做手机，不硬拼参数 |
   | `{QUARANTINE_HTML}` | 隔离区HTML（双层或单层） | `<div class="quarantine"><div class="row"><span class="label">⚠️ 首轮隔离</span><div class="tags"><span class="tag">...</span></div></div></div>`<br>二次迭代用双层：第二行加 `<div class="row"><span class="label">❌ 前轮产出已ban</span><div class="tags"><span class="tag-ban2">...</span></div></div>` |
   | `{CARD_BORDER}` | 单卡上边框渐变色 | `#6366f1, #06b6d4` |
   | `{CARD_ROWS}` | 卡片HTML | `<div class="card card-0">...` |
   | `{INSIGHT_BG_1}` | 底部建议块渐变色1 | `#2a1a44` |
   | `{INSIGHT_BG_2}` | 底部建议块渐变色2 | `#1a2436` |
   | `{INSIGHT_BORDER}` | 底部建议块边框色Hex | `#6d28d944` |
   | `{INSIGHT_HEADING}` | 底部建议块标题色Hex | `#c4b5fd` |
   | `{SUMMARY}` | 底部综合建议块完整HTML | `<div class="key-insight">...` |
   | `{SOURCES}` | 页脚来源标签 | `城市规划/博弈军事/毛选×2/汉语语言学` |

3. **生成卡片内容**：每行表格 → 一个 `.card` 块，结构为：
   - `.card-source`：来源方式 + 领域（如 `🔵 城市规划 · 基础设施先行`）
   - `.its-says`：它说（引用内容，11-13字体）
   - `.you-try`：所以你试试（14字体 + 高亮 `<strong>` 标签）
   - 每张卡需设置 `card-N` class（N=0..5）对应不同渐变上边框

4. **配色方案复用**（在会话内保持一致性，完整色库见 `references/poster-palettes.md`）：

   主题  | 徽章色 | 高亮色 | 渐变色
   ------|--------|--------|-------
   默认紫蓝 | #6366f1 | #22d3ee | #38bdf8 → #a78bfa → #f472b6
   黄黑   | #f59e0b | #fbbf24 | #a78bfa → #60a5fa → #34d399
   青色   | #06b6d4 | #22d3ee | #38bdf8 → #a78bfa → #f472b6

5. **生成&预览**：取 `skill_view('wildidea', 'templates/poster.html')` 返回的HTML → 将所有 `{PLACEHOLDER}` 替换为实际内容（用 Python `str.replace()` 或 `jinja2`）→ 写为 `/tmp/wildidea-xxx.html` → `cd /tmp && python3 -m http.server 789X`（background）→ `browser_navigate` → 截图

6. **截图发送**：`browser_vision` 在 DeepSeek 上会失败（返回screenshot_path），直接取缓存路径 → `MEDIA:/path/to/screenshot.png` 发微信

7. **清理**：`process(action='kill', session_id=...)` 杀掉后台 http.server

**设计约定**：暗色主题（#0c0c0e 底色），800px 宽，2列卡片网格布局，底部综合建议用紫色渐变圆角大色块。每张卡片顶部有3px渐变色带作为视觉区分。

## 已知问题 / Pitfalls

1. **方法B（随机组字）可能出"弱"结果** — 如"着扬"只搜到汉语语法，domain 难提取。此时如实标注对应语言学领域即可，不必重试或编造。
2. **browser_vision 在部分模型上不可用** — 如 DeepSeek 不支持 `image_url` 格式，vision 分析会返回 400 错误。此时直接使用 `browser_vision` 调用失败返回的 `screenshot_path` 作为图片源即可（截图实际已成功保存）。
3. **海报模板需完整替换所有占位符** — 遗漏任何一个 `{PLACEHOLDER}` 会导致页面显示原始字符串。预览后用 `browser_navigate` 加载验证。
4. **http.server 端口抢占** — session 内多次做海报时用递增端口（7890→7891→7892...）避免端口冲突。
5. **联网验证循环可能耗尽所有范式** — 一个远域最多重试3次范式映射。全部用完仍被ban → 如实标注🚫跳过，不编造不硬凑。
6. **搜索词构造** — 搜索词用[核心动作短语] + [用户领域]，如"手机 场景意图识别 AI 自动识别拍照场景"。如果结果出现产品发布/专利/论文/媒体报道 → 判定为已有方案。如果只出现无关内容或理论讨论 → 判定为新方案通过。
7. **舒适度检查的主观性** — 严格标准："发到内部评审群会有人反对吗？" 如果犹豫 → 算不通过。宁可减少产出条数也不放行"有点合理"的答案。

> 纪律：先异化，再翻译。先让远域说话，再让用户领域回应。不行就重抽，直到够野够新。

# 机制迁移审查

用于算法、模型、识别、预测、优化、训练、推理、信号处理、搜索、推荐或控制问题。

目标：把候选从反大路货生成器推进到源域优先的机制迁移器。先锁定源域，再映射用户领域，再做反马后炮审查。算法/科研类标准模式必须在线粗查论文、代码、benchmark、专利和数据集；输出只代表独立判官评分和联网粗查通过，文献级 novelty audit 必须标为待人工验证。

## 模式选择

| 模式 | 使用条件 | 评分重心 |
|------|----------|----------|
| 算法技术迁移 | 用户需要可快速实现的算法模块 | 可实现性、实现清晰度、与 baseline 可区分 |
| 远域机制迁移 | 用户要更野、更非共识的机制方向 | 源域独立性、机制新意、反包装能力 |

每轮算法类输出可混合两种模式，但必须在统计里记录各自数量。

## 源域锁定

生成对应物之前，先内部完成源域机制冻结：

1. 只写源域，不写用户领域词。
2. 不写用户领域常规方法名、模块名、数据集名和任务名。
3. 抽取源域机制的输入、状态、输出、反馈。
4. 标出触发条件、阈值、窗口、轮次、边界或失败条件。
5. 给每个源域机制原型分配编号。
6. 冻结后再映射到用户领域变量。

如果源域机制冻结阶段已经出现用户领域答案，丢弃并重抽。

## 目标域禁词表

每轮先建立目标域禁词表，来自：
- 用户输入里的任务名、数据类型、指标和对象。
- V0 常规方向。
- V1 上轮产出。
- 目标领域常规模块、训练策略、数据处理方式和候选名称。

源域锁定阶段禁止出现禁词表内容。映射阶段只能引用已冻结的原型编号，不得新增或改写源域机制。

## 原型编号

每个源域机制原型使用 `P01`、`P02` 递增编号。最终表必须保留编号。

编号规则：
- 一个编号只绑定一个源域机制。
- 映射时不得修改编号对应的输入、状态、输出、反馈。
- 如果映射时发现机制不适配，丢弃该编号并重抽，不要现场改机制。

## 源域冻结日志

每轮算法类任务必须内部保留源域冻结日志：

```markdown
| 原型ID | 来源 | 源域机制原文 | I/S/O/F | 边界 | 目标域禁词命中 |
```

默认不展示日志；用户要求审查、复盘或质疑马后炮时，先展示日志再展示映射表。若日志缺失，该轮输出不可信，必须重跑。

每个最终候选必须展示一个“源域原型/外域抽象结果”。这是映射到用户领域前的最后阶段，写成通用操作机制，不出现用户领域术语。例如：
- 源域机制原文：Kalman filter 用预测值和观测值的残差更新状态。
- 源域原型/外域抽象结果：每次新证据进入时，先比较预测和观测的偏差，再按偏差大小调整信任。

## 映射字段

每条候选必须能填满这些字段：

| 字段 | 要求 |
|------|------|
| 原型ID | 已冻结的源域机制编号 |
| 源域机制原文 | 带边界的源域机制，不是领域名 |
| 源域输入/状态/输出/反馈 | 四项至少三项明确 |
| 源域原型/外域抽象结果 | 映射前冻结出的通用操作机制，不能出现用户领域术语 |
| 用户领域怎么干 | 用通俗操作语言说明：拿什么数据，按什么顺序做什么，什么条件触发，触发后模型、样本或训练流程怎么变 |
| 改变的算法环节 | 训练、推理、采样、标注、校准、路由、损失、评估或部署中的一个 |
| novelty 类型 | 评估协议、训练目标、表征结构、推理机制、部署策略、数据准入、采样策略、校准机制、路由机制、损失函数 |
| 去锚点成立等级 | 不成立、部分成立、成立 |
| 去锚点退化物 | 去掉源域机制后最接近的普通做法 |
| 最近邻相同点 | 与最近方法共享的任务、变量、模块或训练口径 |
| 最近邻不同点 | 源域机制带来的结构差异 |
| 被吃掉风险 | 低/中/高；高则淘汰 |
| 最强反驳 | 审稿人视角下，为什么它可能只是已有方法换名 |
| 最小可证伪实验 | 一个能失败的最小对照实验 |
| 源域依赖度 | 无源域机制时是否难以自然产生；低/中/高 |
| 结构保真度 | 源域输入、状态、输出、反馈是否完整迁移；低/中/高 |
| 本域退化风险 | 是否容易被解释成目标领域已有方法换名；低/中/高 |

## 三层验证

查重状态只允许三档（搜索证据必须写入 `outputs/<topic>.search.json` 供 `validate_poster.py --search-sidecar` 校验）：
- 联网粗查已启用：标准模式默认状态。算法/科研类必须查论文、代码、benchmark、专利和数据集说明，写”联网粗查已启用；文献级 novelty audit 待人工验证”。
- 文献集严格查重已启用：用户提供 BibTeX、DOI、Zotero、标题摘要 CSV、PDF 文件夹或明确本地文献集，必须记录命中论文、相同机制、不同机制和淘汰判断。
- 用户要求跳过查重：仅限快速/草稿/不要联网，不得标为标准模式。

1. 词面查重：查同名方法、同名模块、同名训练策略。
2. 机制查重：查不同名字但同输入、同状态、同输出、同反馈的实现。
3. 退化查重：去掉外域机制后，看是否退化为已有常规方法。
4. 最近邻差异审查：写清相同点、不同点、被吃掉风险。
5. 反方审查：写出最强反驳，判断是否还能保留机制差异。

生成 HTML 时，把本题的目标域禁词表（任务名、数据类型、对象、指标、常规方法名）作为 `--forbid-proto-term` 传给 `validate_poster.py`，自动卡住 `.proto`（源域原型/外域抽象结果）里残留的用户领域术语。`.proto` 若含禁词说明源域机制没有真正冻结、被目标域提前吸收，必须重写。

判定规则：
- 词面或机制直接命中：淘汰。
- 去锚点成立等级为“不成立”：可保留。
- 去锚点成立等级为“部分成立”：仅当最近邻不同点清楚、被吃掉风险不高时保留。
- 去锚点成立等级为“成立”：淘汰或降级为弱迁移，不进标准模式最终 10 条。
- 退化后仍有明确机制差异：可保留。
- 退化后只剩命名差异、轻微后处理或普通组合：淘汰。
- 最近邻不清楚：标为搜索不可判定，不进入最终 10 条。
- 被吃掉风险高或本域退化风险高：淘汰。
- 最强反驳无法回应：淘汰。

淘汰/重抽记录：
- 每次丢弃候选时，记录来源、候选名或查询词、淘汰原因、重抽去向。
- 最终输出必须展示淘汰/重抽记录。若超过 8 条，只展示最关键 8 条并给总数。
- 搜索不可判定、随机组词无真实机制、去锚点成立、最近邻风险高都必须进入记录。

## V1 Ban 分类

上一轮输出不要全部硬 ban，按四类处理：

| 类型 | 处理 |
|------|------|
| Concept-ban | 禁止重复同一概念名或命名壳 |
| Mechanism-ban | 禁止重复同一输入/状态/输出/反馈结构 |
| Module-ban | 禁止重复同一主创新算法位置 |
| Soft-ban | 可作为组件复用，但不得作为主创新再次输出 |

## 输出纪律

- 不写长解释。
- 不把验证写成“未命中所以新”。
- 输出表里不要把用户领域映射写成变量名清单。
- 每条必须展示“源域原型/外域抽象结果”，位置在源域机制原文之后、用户领域怎么干之前。
- “用户领域怎么干”必须通俗、具体、可想象，可以比其他列更长。
- 该列必须包含：拿什么数据、做什么动作、什么条件触发、触发后改变什么。
- 源域冻结日志只在用户要求审查、严格版或日志时展示。
- 必须写最近邻和退化物。
- 必须保留源域冻结日志；日志缺失则重跑。
- 必须写去锚点成立等级：不成立、部分成立、成立。
- 最近邻必须写相同点、不同点和被吃掉风险。
- 必须写最强反驳。
- 必须写 novelty 类型，并限定在：评估协议、训练目标、表征结构、推理机制、部署策略、数据准入、采样策略、校准机制、路由机制、损失函数。
- 牵强度不得只写通过；改写为源域依赖度、结构保真度、本域退化风险。
- 算法技术迁移和远域机制迁移分别交给独立判官评分，不合并成一个分数。
- 最小可证伪实验必须包含对照对象和失败判据。
- 统计不得写需人工验证0条；算法/科研类必须写“独立判官通过N条；查重状态：联网粗查已启用/文献集严格查重已启用；文献级 novelty audit 待人工验证N条”。只有快速草稿可写“用户要求跳过查重”。

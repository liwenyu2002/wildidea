# 搜索集成口径

标准模式强制联网搜索，不绑定具体工具，但必须调用当前 Agent/运行环境真实可用的联网搜索能力。不得只用模型记忆、常识判断或离线联想充当查重。

## 搜索工具兜底链

联网搜索按以下优先级尝试，找到可用的就用，全部不可用时走兜底：

1. **首选：当前 Agent 的联网搜索能力**（如 WebSearch、browser search、WebFetch、联网 MCP、插件搜索或宿主环境提供的搜索工具）。标准模式必须优先调用这一层，记录搜索词、命中标题/链接和判定。
2. **次选：`scripts/search_helper.py`**（搜狗引擎，零 API key，Python 标准库 + curl 双保险）
   - 基本搜索：`python scripts/search_helper.py "搜索词" --top 5`
   - 带抓取第一条正文：`python scripts/search_helper.py "搜索词" --top 3 --fetch-first`
   - 该脚本是 helper fallback。若它失败，不得把失败结果写成查重通过。
3. **再次确认：询问用户当前环境是否还有可用联网搜索工具**（MCP server、WebSearch、WebFetch、其他搜索 API 等），直接用用户指定的工具。
4. **无联网能力时停止标准模式**：当所有联网搜索手段都不可用时，不得继续标准模式。必须询问用户是否开启搜索能力或切换为快速/草稿模式；草稿模式最终必须标注"联网查重未启用，结果仅供发散"。

## 学术查重（OpenAlex，arXiv 兜底）

目标领域新颖度检查（`core/novelty_check.py` 的 `check_target_field_novelty`，对应 SKILL.md/wildidea-skill.md 中的 Target Field Novelty Check）按题型分流检索源，不再统一走搜狗；是否运行由独立的查重开关决定，不再要求 `risk_profile == "research"`，具体生效条件（`Config.research_novelty_check` 三态语义、CLI `--novelty-check`/`--no-novelty-check`、网页“是否查重”开关）见 wildidea-skill.md 该节“生效条件”一条：

- **算法/研究题默认走 OpenAlex 学术检索**（`https://api.openalex.org/works`，约 2.5 亿篇文献、免 API key、相关性排序、结构化 JSON）：搜狗这类通用网页检索对判断"该具名方法用于该研究问题的具体组合是否已有发表工作"这种学术新颖性问题信号太弱。OpenAlex 覆盖远比 arXiv 预印本更全、可达性也更稳；解析出命中论文的标题与摘要（由 abstract_inverted_index 还原）作为证据交给判官模型。arXiv（`https://export.arxiv.org/api/query`，Atom XML，仅预印本）保留为可显式指定的兜底后端。中文问题先由一次轻量 LLM 调用翻译成英文检索词，再做 AND 连接查询，避免中文词在英文库零命中。
- **产品/策略题仍走通用网页检索**（搜狗兜底链，见上一节），不切换到学术库——这类问题要查的是产品发布、功能公布、竞品案例，不是学术文献。
- **判定口径**：只在证据显示"该方法用于该问题"这个具体组合已是既有发表工作时，才判定"领域内已知"（`is_known=True`）。检索命中很多与该方法相关的论文，但没有一篇是把它用在这个目标问题/领域上——例如该方法在其原生领域很有名、广泛发表，但没人把它用来解这个具体问题——不算已知；只有相关论文这一条本身不构成判定依据。
- **可插拔**：`check_target_field_novelty` 的 `_search_backend` 形参不受此默认分流影响——auto-research 等集成方仍可注入自己的检索后端（内部论文库、Semantic Scholar 等）整体替换默认的 OpenAlex/搜狗组合，接口签名不变。这是查重稳定性与权威性的真正上限：内置 OpenAlex 是免 key 的尽力而为默认，注入自有文献库后不再依赖公网学术 API 的可达性。
- **降级纪律**：OpenAlex/arXiv 请求失败、超时、限流或解析失败时，先 urllib 后 curl 兜底；仍失败则和搜狗兜底链一样优雅降级为"未知"（`is_known=False`），不得因为学术检索工具故障就误伤一个本来合格的候选。

## 查重驱动换卡

查重开关生效时（不再要求 `risk_profile == "research"`，见上一节“生效条件”），目标领域新颖度检查判定某候选"领域内已知"（`is_known=True`）后，不再只是被动打标签展示——这条检查从过滤器升级为主动的新颖度驱动器：

- 该槽位换一个**新的源锚点**（不是同一锚点改措辞，也不是同一机制重新表述）重新走完整流程：质量门槛 -> 独立判官 -> 再跑一次目标领域新颖度检查。
- 换锚点重抽最多 `max_novelty_rerolls` 次，与 Web-Parity Contract/Web v1.4 Parity 里质量门槛与判官阶段最多 2 次重抽的预算相互独立计数，不共用同一上限。
- 每次重新检查都各自记一次 `novelty_flag` 事件，便于追踪一个槽位为摆脱"已知"标记换了几次锚点、换到了哪些锚点。
- 达到 `max_novelty_rerolls` 上限仍判定"领域内已知"时，保留最后一版并如实标注"领域内已知"，同时披露已耗尽换锚点重抽额度——不得无限换锚点，也不得在耗尽额度后把它悄悄标成通过。
- 目的：让最终交付、以及喂给 auto-research 的 idea 尽量都不与已发表工作撞车，而不是把撞车的候选原样交出去只贴一个标签了事。

## 三种用途

1. **随机组字撞域**（最大化随机，不做过滤）：为随机生成的两字组合找到一个真实外部语境。流程：随机抽两个字（不要求组合有意义，越无意义搜索结果越不可预期）→ 联网搜索该词（按上述兜底链）→ 拿搜索结果第一条 → 从中提取一种方法论、机制、规则或边界 → 映射到用户领域。不要因为组合"看起来没意义"就重抽；但如果无法联网找到外部语境，该随机组词候选不得进入标准结果。
2. 联网验证：判断某个用户领域对应物是否已有明确实现。
3. 论文/专利/代码粗查：算法、科研和高技术问题默认启用，判断候选是否已有同名或同机制实现。

算法类问题必须优先查论文、开源仓库、benchmark、专利和数据集说明；同时记录词面查重、机制查重、退化查重和最近邻差异审查。

## 三档查重状态

| 状态 | 使用条件 | 写法 |
|---|---|---|
| 联网粗查已启用 | 标准模式默认；产品/策略查产品、功能、专利、竞品；算法/科研查论文、代码、benchmark、专利 | `联网粗查已启用；文献级 novelty audit 待人工验证` |
| 文献集严格查重已启用 | 用户提供 BibTeX、DOI、Zotero、标题摘要 CSV、PDF 文件夹或明确本地文献集 | `文献集严格查重已启用；记录最近邻命中依据` |
| 用户要求跳过查重 | 仅限快速/草稿/不要联网，不得标为标准模式 | `快速草稿；联网查重未启用` |

## 搜索证据 sidecar 格式

标准模式下，每条最终候选的搜索记录写入 `outputs/<topic>.search.json`，由 `validate_poster.py --search-sidecar` 校验。

```json
{
  "candidates": [
    {
      "id": "P01",
      "name": "候选名",
      "source": "来源机制名",
      "searches": [
        {
          "query": "搜索词",
          "status": "found | no_result | failed",
          "hits": [
            {"title": "结果标题", "url": "结果链接"}
          ],
          "decision": "pass | ban | needs_manual_check"
        }
      ],
      "decision": "pass | ban | needs_manual_check",
      "note": "联网粗查已启用；文献级 novelty audit 待人工验证"
    }
  ],
  "meta": {
    "mode": "standard",
    "search_enabled": true,
    "date": "2026-05-31"
  }
}
```

字段说明：
- `id`：候选编号（与海报卡片 `.id` 对应）
- `name`：候选机制名
- `source`：来源机制具体名称（如 `CTC blank 折叠`、`Thompson sampling`）
- `searches`：该候选的搜索记录数组，至少 1 条
- `searches[].status`：`found`（有结果）、`no_result`（无结果）、`failed`（搜索工具不可用）
- `searches[].decision`：单条搜索的判定
- `decision`：该候选的最终判定
- `meta.mode`：`standard`（标准模式）或 `quick`（快速模式）
- `note`：查重状态标注
- 标准模式下，`searches[]` 必须来自真实联网调用；不得把模型凭记忆写出的“可能已有/未发现”塞进 sidecar。

## 判定规则

判定为 `ban`：
- 明确产品发布。
- 明确功能公布。
- 专利、论文、品牌案例直接命中。
- 竞品已有同名或同机制实现。
- 算法类问题中，用户领域已有同任务、同机制、同训练/验证口径的论文或代码。
- 算法类问题中，去掉源域机制后只剩已有常规方法。
- 算法类问题中，最近邻相同点覆盖主要机制，且不同点只剩命名或轻微后处理。
- 算法类问题中，最强反驳显示该候选只是已有方法换名且无可测机制差异。

判定为 `pass`：
- 无结果。
- 结果只是论坛讨论、用户反馈、理论文章。
- 命中内容与待验证对应物不是同一实现。

判定为 `needs_manual_check`：
- 搜索工具不可用（已按兜底链尝试所有选项后仍不可用）。
- 超时、反爬、权限错误。
- 结果含混，无法判断是否已有实现。

标准模式中 `needs_manual_check` 不得标为通过或未达标保底。该槽位仍按“初次生成 + 最多 2 次重抽”执行；上限后没有可确认候选则保留失败状态，不得无限换候选。

算法/科研类若只是尚未完成文献级 novelty audit，不要记为 `needs_manual_check`；应在最终统计中单独标记为"待人工验证"。

## 纪律

- 搜索失败时不凭常识补结果，且标准模式不得让该候选进入最终列表。
- 不因出现任意搜索结果就淘汰。
- 不把搜索摘要扩写成论证段。
- 最终输出只写判定，不贴长搜索内容。

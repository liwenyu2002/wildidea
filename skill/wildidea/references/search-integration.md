# 搜索集成口径

标准模式默认强制联网搜索，不绑定具体工具。优先使用当前环境中最稳定、最少噪音的搜索方式。

## 搜索工具兜底链

联网搜索按以下优先级尝试，找到可用的就用，全部不可用时走兜底：

1. **首选：`scripts/search_helper.py`**（搜狗引擎，零 API key，Python 标准库 + curl 双保险）
   - 基本搜索：`python scripts/search_helper.py "搜索词" --top 5`
   - 带抓取第一条正文：`python scripts/search_helper.py "搜索词" --top 3 --fetch-first`
   - 该脚本在当前环境直接可用，无需注册或配置。
2. **次选：询问用户当前环境有哪些可用的联网搜索工具**（MCP server、WebSearch、WebFetch、其他搜索 API 等），直接用用户指定的工具。
3. **兜底：直接联想**——当所有联网搜索手段都不可用时，基于模型自身知识生成候选，但在最终统计中必须标注"联网搜索未启用，结果仅基于模型自审"，且不得标为标准模式。

## 三种用途

1. **随机组字撞域**（最大化随机，不做过滤）：为随机生成的两字组合找到一个真实外部语境。流程：随机抽两个字（不要求组合有意义，越无意义搜索结果越不可预期）→ 联网搜索该词（按上述兜底链）→ 拿搜索结果第一条 → 从中提取一种方法论、机制、规则或边界 → 映射到用户领域。不要因为组合"看起来没意义"就重抽。
2. 联网验证：判断某个用户领域对应物是否已有明确实现。
3. 论文/专利/代码粗查：算法、科研和高技术问题默认启用，判断候选是否已有同名或同机制实现。

算法类问题必须优先查论文、开源仓库、benchmark、专利和数据集说明；同时记录词面查重、机制查重、退化查重和最近邻差异审查。

## 三档查重状态

| 状态 | 使用条件 | 写法 |
|---|---|---|
| 联网粗查已启用 | 标准模式默认；产品/策略查产品、功能、专利、竞品；算法/科研查论文、代码、benchmark、专利 | `联网粗查已启用；文献级 novelty audit 待人工验证` |
| 文献集严格查重已启用 | 用户提供 BibTeX、DOI、Zotero、标题摘要 CSV、PDF 文件夹或明确本地文献集 | `文献集严格查重已启用；记录最近邻命中依据` |
| 用户要求跳过查重 | 仅限快速/草稿/不要联网，不得标为标准模式 | `快速草稿；联网查重未启用` |

## 搜索证据 sidecar 格式

标准模式下，每条最终候选的搜索记录写入 `outputs/<topic>.search.json`，由 `validate_poster.py --search-sidecar` 校验。

```json
{
  "candidates": [
    {
      "id": "P01",
      "name": "候选名",
      "source": "来源机制名",
      "searches": [
        {
          "query": "搜索词",
          "status": "found | no_result | failed",
          "hits": [
            {"title": "结果标题", "url": "结果链接"}
          ],
          "decision": "pass | ban | needs_manual_check"
        }
      ],
      "decision": "pass | ban | needs_manual_check",
      "note": "联网粗查已启用；文献级 novelty audit 待人工验证"
    }
  ],
  "meta": {
    "mode": "standard",
    "search_enabled": true,
    "date": "2026-05-31"
  }
}
```

字段说明：
- `id`：候选编号（与海报卡片 `.id` 对应）
- `name`：候选机制名
- `source`：来源机制具体名称（如 `CTC blank 折叠`、`Thompson sampling`）
- `searches`：该候选的搜索记录数组，至少 1 条
- `searches[].status`：`found`（有结果）、`no_result`（无结果）、`failed`（搜索工具不可用）
- `searches[].decision`：单条搜索的判定
- `decision`：该候选的最终判定
- `meta.mode`：`standard`（标准模式）或 `quick`（快速模式）
- `note`：查重状态标注

## 判定规则

判定为 `ban`：
- 明确产品发布。
- 明确功能公布。
- 专利、论文、品牌案例直接命中。
- 竞品已有同名或同机制实现。
- 算法类问题中，用户领域已有同任务、同机制、同训练/验证口径的论文或代码。
- 算法类问题中，去掉源域机制后只剩已有常规方法。
- 算法类问题中，最近邻相同点覆盖主要机制，且不同点只剩命名或轻微后处理。
- 算法类问题中，最强反驳显示该候选只是已有方法换名且无可测机制差异。

判定为 `pass`：
- 无结果。
- 结果只是论坛讨论、用户反馈、理论文章。
- 命中内容与待验证对应物不是同一实现。

判定为 `needs_manual_check`：
- 搜索工具不可用（已按兜底链尝试所有选项后仍不可用）。
- 超时、反爬、权限错误。
- 结果含混，无法判断是否已有实现。

标准模式中 `needs_manual_check` 不进入最终 10 条。除非用户明确要求快速/草稿/不要联网，否则继续换候选或说明无法完成标准模式。

算法/科研类若只是尚未完成文献级 novelty audit，不要记为 `needs_manual_check`；应在最终统计中单独标记为"待人工验证"。

## 纪律

- 搜索失败时不凭常识补结果。
- 不因出现任意搜索结果就淘汰。
- 不把搜索摘要扩写成论证段。
- 最终输出只写判定，不贴长搜索内容。
